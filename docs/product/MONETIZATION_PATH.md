# Monetization Path

## Назначение документа

Документ фиксирует реалистичный путь к ранним monetizable outcomes внутри `personal-lms` и связывает его с учебным pipeline.

## Что зафиксировано

- Early monetization рассматривается как допустимая и желательная прикладная цель.
- Ценность создается через маленькие рабочие результаты, а не через "завершение теории".
- Главная практическая ветка текущего этапа: Telegram / automation / AI utility tools.
- Крупные блоки курса должны завершаться checkpoint artifacts, а не только закрытыми уроками.

## Принцип ранней прикладной ценности

Ранние результаты должны быть:

- небольшими по объему;
- проверяемыми по результату;
- понятными для демонстрации клиенту/работодателю;
- совместимыми с текущим уровнем learner на ранних этапах.

## Реалистичные типы мини-проектов

- automation scripts для регулярных рутин;
- integration utilities между 2–3 сервисами;
- Telegram notification/intake bot;
- FastAPI + SQLite microservice;
- webhook/service patterns для событийной обработки;
- AI extraction / AI triage utilities;
- client-facing micro-MVP на узкую задачу.

## Почему именно эти типы

Они дают лучший баланс между:

- скоростью сборки;
- видимой полезностью;
- технической сложностью, посильной в 6-месячном окне;
- возможностью повторного использования в портфолио.

## Связь с курсом

Путь monetization встроен в pipeline:

1. automation scripts — первый слой полезности;
2. integration utilities — добавление внешних данных и API;
3. Telegram bot — пользовательский вход/оповещения;
4. FastAPI + SQLite microservice — стабильный backend-контур;
5. webhook/service patterns — событийные workflow;
6. AI extraction/triage — прикладное AI-усиление;
7. client-facing micro-MVP — упаковка в демонстрируемый продукт.

## Checkpoint → portfolio → sellable linkage

На границе крупных блоков должен быть checkpoint project с проверяемым definition of done.

Требуемая цепочка результата:

1. **Checkpoint artifact**: интеграционный учебный результат блока.
2. **Portfolio artifact**: тот же результат оформлен для демонстрации (GitHub + краткий кейс).
3. **Sellable artifact**: при достаточном качестве и стабилизации превращается в небольшое клиентское предложение или micro-service utility.

Таким образом, курс оценивает прогресс по демонстрируемым и проверяемым выходам, а не по количеству завершенных уроков.

## Portfolio-friendly artifacts

Каждый этап должен давать артефакт, который можно:

- разместить в GitHub;
- описать в кратком кейсе (проблема → решение → результат);
- показать на собеседовании или клиентском разговоре;
- использовать как базу для следующего checkpoint-проекта.

## Ограничения и anti-scope

Документ не фиксирует:

- финансовые обещания;
- гарантированные сроки дохода;
- стратегию масштабирования в агентство/стартап.

Фокус — на учебно-прикладных результатах раннего уровня.

## Влияние на следующие этапы

- При authoring заданий приоритет получают сценарии с реальным utilitarian outcome.
- Checkpoint-проекты оцениваются по применимости, демонстрируемости и готовности к портфолио/упаковке.
- Curriculum hardening должен усиливать путь automation → integrations → backend → AI utility → micro-MVP.
