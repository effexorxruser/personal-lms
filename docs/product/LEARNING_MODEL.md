# Learning Model

## Назначение документа

Документ фиксирует учебную логику `personal-lms`: как пользователь движется по маршруту, как фиксируется progress и почему основной принцип — execution, а не consumption.

## Главный принцип

Обучение должно вести к выполнению.

Материал считается полезным только тогда, когда он связан с действием:

- прочитать/посмотреть нужный источник;
- выполнить задачу;
- отправить результат;
- получить review;
- закрыть шаг по definition of done.

## Маршрут

Маршрут должен давать понятный next step.

Пользователь не должен каждый раз выбирать из десятков материалов. Система должна показывать:

- текущий курс;
- текущий модуль;
- следующий урок;
- связанную задачу;
- статус progress.

## Progress

Progress нужен не как декоративный процент, а как runtime state обучения.

Минимально progress должен отвечать:

- какие уроки доступны;
- какие уроки начаты;
- какие уроки завершены;
- где пользователь застрял;
- какой следующий шаг логичен.

## Tasks

Каждый важный учебный шаг должен вести к задаче.

Задача должна быть достаточно маленькой, чтобы ее можно было выполнить, но достаточно практичной, чтобы она создавала навык.

Примеры типов задач:

- написать небольшой Python-скрипт;
- пройти CLI/Git упражнение;
- написать тест;
- сделать SQL-запрос;
- реализовать endpoint;
- собрать маленький CRUD fragment;
- добавить простую AI-enabled функцию.

Конкретные задачи фиксируются в content layer и могут уточняться.

## Submissions

Submission — это результат выполнения задачи.

Формат submission может включать:

- текстовый ответ;
- ссылку на commit/repository;
- файл;
- вывод команды;
- другой проверяемый артефакт.

Точный формат для всех типов задач не полностью зафиксирован.

## Review

Review нужен, чтобы пользователь получал обратную связь и не проходил маршрут формально.

Review может проверять:

- выполнена ли задача;
- соответствует ли результат definition of done;
- есть ли ошибки;
- что улучшить;
- можно ли идти дальше.

AI-assisted review возможен позже, но не должен превращаться в autopilot, который размывает фиксированный курс.

## Checkpoint projects

Checkpoint project — это контрольный артефакт после части маршрута.

Его задача:

- собрать несколько навыков в один маленький рабочий результат;
- показать применимость знаний;
- создать материал для портфолио;
- зафиксировать переход от темы к практике.

Конкретный список checkpoint projects не полностью зафиксирован.

## Stuck flow

Stuck flow нужен для моментов, когда пользователь не может завершить шаг.

Минимальная логика:

- пользователь отмечает, что застрял;
- система фиксирует место;
- предлагает уточнить проблему;
- помогает вернуться к источнику, подсказке или review;
- не ломает маршрут и не перепрыгивает через execution.

## Weekly recap

Weekly recap должен помогать поддерживать регулярность.

Он может фиксировать:

- что было завершено;
- какие задачи были отправлены;
- где были stuck moments;
- какой следующий фокус на неделю.

Точный формат weekly recap может быть уточнен позже.

## Definition of done для урока

Урок не считается завершенным только потому, что пользователь открыл страницу.

Минимальный done должен включать:

- источник изучен в указанном объеме;
- связанная задача выполнена, если она есть;
- submission отправлен, если требуется;
- review не блокирует переход дальше;
- пользователь понимает, какой навык был закрыт.

## Не consumption, а execution

Платформа должна защищать от tutorial hell.

Запрещенный паттерн: бесконечно смотреть уроки без результата.

Целевой паттерн: маленькие, регулярные, проверяемые действия, которые постепенно собираются в реальные backend + AI product skills.
