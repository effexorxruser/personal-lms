# Authoring Model

## Назначение документа

Документ фиксирует модель загрузки и жизненного цикла реального curriculum content в `personal-lms`.

## Что зафиксировано

- Контент хранится в файлах, runtime state хранится в БД.
- Учебная единица реализуется как lesson wrapper поверх curated source.
- Исполнение фиксируется через task → submission → review.
- Крупные участки маршрута закрываются checkpoint project.

## Базовые принципы

### 1) Content in files / runtime state in DB

В файлах хранятся:

- структура курса;
- тексты wrappers;
- формулировки задач;
- definition of done;
- checkpoint-описания.

В БД хранятся:

- статусы прохождения;
- submissions;
- review результаты;
- stuck и recap runtime-данные.

### 2) Lesson wrapper model

Каждый lesson должен содержать:

- цель шага;
- curated source(ы) и объем прохождения;
- обязательное действие;
- критерий done;
- переход к следующему шагу.

### 3) Task layer

Task — обязательный execution-слой урока (кроме явно отмеченных редких обзорных уроков).

Task должен быть:

- проверяемым;
- ограниченным по объему;
- привязанным к навыку текущего этапа;
- оформленным так, чтобы результат можно было отправить как submission.

### 4) Submission / review model

Submission обязан иметь проверяемый формат (ссылка, файл, текст, команда/вывод или их комбинация).

Review обязан фиксировать:

- pass/fail или эквивалентный статус;
- комментарий по качеству;
- что исправить при необходимости;
- условие перехода дальше.

### 5) Checkpoint project concept

Checkpoint проект обязателен на границе крупных блоков и собирает несколько навыков в один портфолио-дружественный результат.

## Ожидаемая структура контента

### Block

- цель блока;
- список модулей;
- ожидаемый block-level artifact;
- checkpoint-критерий на выходе.

### Module

- цель модуля;
- список уроков;
- dependency на предыдущие шаги;
- module-level outcome.

### Lesson

- lesson wrapper;
- source references;
- task;
- definition of done;
- hints/stuck entry.

### Task

- постановка;
- входные условия;
- ожидаемый результат;
- формат submission;
- оценочный критерий review.

### Checkpoint

- интеграционная задача блока;
- минимальные функциональные требования;
- критерии качества;
- артефакт для портфолио/демо.

## Обязательные требования к authoring реального curriculum

При загрузке реального контента обязательны:

- явная привязка к pipeline-этапу;
- указание core/supplement источников;
- наличие execution-шага и проверяемого outcome;
- определенный submission формат;
- критерии review и done;
- связь с checkpoint-логикой крупного блока.

Контент без этих элементов не считается authoring-ready.

## Что не входит / что отложено

- Полная детализация всех будущих типов review automation.
- AI-autopilot генерация курса без фиксированного маршрута.
- Сложные multi-course абстракции до завершения first real curriculum pass.

## Влияние на следующие этапы

- Authoring readiness проверяется по этому документу.
- Curriculum onboarding должен отклонять контент, который не выполняет обязательные требования.
- Runtime развитие должно поддерживать эту модель, а не обходить ее.
