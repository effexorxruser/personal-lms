# LAIN CURRICULUM READINESS

## Назначение

Документ фиксирует readiness-план для Lain Helper в контексте curriculum learning.

Текущий статус: Lain MVP уже есть, но нижеописанные режимы пока не являются полностью продуктово-зафиксированными для массовой генерации контента.

## Принципы

- Lain не заменяет learner execution.
- Lain не превращается в autopilot-генератор решений.
- Ответы должны быть русскоязычными с техническим английским в управляемом формате.

## Обязательные будущие режимы

### 1) `explain_english`

**Цель:** объяснять технический английский в контексте урока/задачи.

Ожидаемое поведение:
- перевод ключевых терминов на русский;
- краткое объяснение контекста использования;
- 1-2 примера употребления в backend/AI задачах.

### 2) `explain_error`

**Цель:** разбор ошибки из terminal/task submission без выдачи полного решения.

Ожидаемое поведение:
- понятная расшифровка причины;
- шаги диагностики;
- безопасная следующая попытка.

### 3) `hint_without_solution`

**Цель:** дать learner-у следующий шаг, не раскрывая финальный код.

Ожидаемое поведение:
- направляющая подсказка;
- ссылки на релевантные source_ids;
- анти-спойлер правило (без готового итогового патча).

### 4) `review_submission`

**Цель:** учебный review ответа learner-а по definition of done.

Ожидаемое поведение:
- что выполнено / что не выполнено;
- где нарушен DoD;
- конкретный actionable следующий шаг.

### 5) `source_summary`

**Цель:** кратко пересказывать разрешенный источник для конкретного lesson/task.

Ожидаемое поведение:
- краткий summary секции источника;
- связь с текущим модулем;
- фиксированный формат ссылок на source id.

## Ready / Not ready граница

### Ready сейчас

- MVP helper внутри lesson scope;
- базовые контекстные ответы;
- поддержка текущего execution-first workflow.

### Не ready пока

- формализованные режимы выше как стабильный контракт;
- строгие quality-гейты для каждой подсказки;
- полноценная рубрика авто-review для всех submission types.

## Минимальный план внедрения (без feature expansion в этом PR)

1. Зафиксировать contract интерфейсов режимов.
2. Добавить тестовые сценарии на границы (anti-spoiler, error explanation quality).
3. Включать режимы поэтапно, начиная с `explain_error` и `hint_without_solution`.
