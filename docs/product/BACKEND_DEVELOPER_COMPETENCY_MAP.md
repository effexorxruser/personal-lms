# Backend Developer Competency Map (Python + AI) — для personal-lms

## Назначение документа

Документ фиксирует целевые компетенции 6-месячного маршрута и правила проверки, что learner действительно умеет делать работу руками.

Карта используется как:

- основа для проектирования lessons/tasks/checkpoints;
- критерий review и progression;
- антидот против "теорию прошел, но делать не могу".

---

## Уровни освоения

Для каждой компетенции используется шкала:

- **L0 — Не умеет**: не может выполнить задачу без полного ведения.
- **L1 — С опорой**: выполняет с подсказками/шаблонами.
- **L2 — Самостоятельно**: выполняет типовые задачи без внешней помощи.
- **L3 — Устойчиво**: выполняет стабильно, отлаживает проблемы и объясняет решения.

Минимальная цель выпускного уровня (после Block 6):

- не ниже **L2** по всем core-компетенциям;
- **L3** минимум в 3 практических зонах (Python execution, API backend, testing/quality).

## Компетенции по блокам

## C0 — Technical English for Backend

### Компетенция C0.1: Чтение docs с извлечением действий

- Что умеет: выделяет конкретный action plan из англоязычной страницы.
- Проверка: по doc-фрагменту формулирует шаги "сделай X, проверь Y".
- Артефакт: doc notes + action checklist.

### Компетенция C0.2: Интерпретация ошибок

- Что умеет: объясняет смысл error message и следующий шаг отладки.
- Проверка: разбор минимум 3 реальных ошибок.
- Артефакт: error log с комментариями.

### Компетенция C0.3: Технический словарь

- Что умеет: оперирует базовыми backend/API/AI терминами без полного перевода.
- Проверка: glossary + применение терминов в mini-review.
- Артефакт: personal glossary.

---

## C1 — Environment, Terminal, Git

### Компетенция C1.1: Terminal execution

- Что умеет: запускает команды, ориентируется в файлах, выполняет базовую диагностику.
- Проверка: выполнение заданного command sequence.
- Артефакт: журнал команд/результатов.

### Компетенция C1.2: Git workflow

- Что умеет: branch → commit → push → PR на малых изменениях.
- Проверка: серия атомарных commit и корректный PR-дифф.
- Артефакт: репозиторий с историей изменений.

### Компетенция C1.3: Reproducible setup

- Что умеет: поднимает `.venv`, запускает проект по инструкции.
- Проверка: cold-start запуск по `setup.md`.
- Артефакт: setup guide + проверочный скрипт.

---

## C2 — Python + Automation

### Компетенция C2.1: Python core для прикладных задач

- Что умеет: использует функции, коллекции, модули, обработку исключений.
- Проверка: решение CLI-задачи среднего размера.
- Артефакт: рабочий CLI-скрипт.

### Компетенция C2.2: Работа с данными и файлами

- Что умеет: читает/пишет JSON/CSV/текст, валидирует вход.
- Проверка: pipeline "входной файл → обработка → выходной отчет".
- Артефакт: automation script + output.

### Компетенция C2.3: Базовая наблюдаемость

- Что умеет: добавляет логирование для диагностики выполнения.
- Проверка: лог отражает ключевые этапы и ошибки.
- Артефакт: logging-enabled версия скрипта.

---

## C3 — HTTP + APIs + SQLite

### Компетенция C3.1: HTTP literacy

- Что умеет: корректно использует методы/статусы/ошибки HTTP.
- Проверка: объяснение и применение на примерах реальных запросов.
- Артефакт: HTTP exercise notes.

### Компетенция C3.2: API consumption

- Что умеет: делает запросы, обрабатывает ответы и сетевые сбои.
- Проверка: стабильный fetch + обработка ошибок.
- Артефакт: API client script.

### Компетенция C3.3: SQLite persistence

- Что умеет: проектирует простую схему, сохраняет и читает данные.
- Проверка: импорт данных + выборка отчетов.
- Артефакт: SQLite DB + скрипты операций.

---

## C4 — FastAPI Backend

### Компетенция C4.1: Проектирование endpoint-ов

- Что умеет: строит CRUD API с понятными ресурсами.
- Проверка: набор endpoint-ов соответствует требованиям задачи.
- Артефакт: FastAPI проект.

### Компетенция C4.2: Валидация и ошибки

- Что умеет: валидирует вход и возвращает корректные ответы/ошибки.
- Проверка: негативные сценарии обрабатываются предсказуемо.
- Артефакт: API contract + примеры.

### Компетенция C4.3: Data integration

- Что умеет: связывает API слой и SQLite-хранилище.
- Проверка: CRUD операции реально изменяют состояние БД.
- Артефакт: рабочий сервис + seed data.

---

## C5 — Testing + Debugging + CI

### Компетенция C5.1: Тестирование

- Что умеет: пишет и запускает unit/integration тесты.
- Проверка: тесты покрывают критические сценарии.
- Артефакт: `tests/` и отчет запуска.

### Компетенция C5.2: Debugging discipline

- Что умеет: воспроизводит баг, локализует причину, фиксирует решение.
- Проверка: bug report с шагами reproduce/fix/verify.
- Артефакт: debugging notes.

### Компетенция C5.3: CI automation

- Что умеет: настраивает CI workflow с автопроверками.
- Проверка: pipeline проходит на PR.
- Артефакт: GitHub Actions workflow.

---

## C6 — AI Integration for Backend

### Компетенция C6.1: API integration с OpenAI

- Что умеет: вызывает модель из backend-кода и обрабатывает ответ.
- Проверка: endpoint возвращает валидный AI-результат.
- Артефакт: AI-enabled endpoint.

### Компетенция C6.2: Structured outputs и tool/function calling

- Что умеет: задает структуру ответа и ограничивает действия модели инструментами.
- Проверка: ответ соответствует схеме, tool вызовы контролируемы.
- Артефакт: schema + integration code.

### Компетенция C6.3: Safety + evaluation basics

- Что умеет: применяет moderation и фиксирует мини-eval качества.
- Проверка: есть guardrails и измеримый eval-отчет.
- Артефакт: moderation/eval docs + результаты.

## Definition of Done на уровне компетенции

Компетенция считается закрытой, если одновременно выполнено:

1. Есть проверяемый execution-результат (код/команда/API/output).
2. Есть submission в согласованном формате.
3. Review не содержит блокирующих замечаний.
4. Learner может кратко объяснить решение и trade-off.

Если отсутствует хотя бы один пункт — компетенция остается "в работе".

## Сигналы готовности к "реальной жизни"

Карта считается practically successful, когда learner способен самостоятельно:

- поднять и доработать backend-проект на FastAPI;
- работать с API и SQLite без hand-holding;
- обеспечить базовое качество через тесты/CI;
- добавить ограниченную AI-функцию с контролем структуры и безопасности;
- показать все это в виде портфолио-артефактов.

Именно эти сигналы используются как критерий выхода из tutorial hell в режим реального инженерного исполнения.
